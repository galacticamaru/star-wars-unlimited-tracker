## Critical Issues

### CR-01: N+1 query loop in `getDeckCardsForUser` — up to 2×N DB round-trips

**File:** `src/db/queries/decks.ts:233-251`

**Issue:** `resolvePrinting` is called inside a `for…of` loop over every deck row (Step 4). Each call fires an independent `await db.select()`. A user with 10 decks each having a leader and a base will issue up to 20 sequential DB queries in a single request. For the current single-user app this is merely slow; if multi-user is ever enabled or the deck count grows, this becomes a correctness-adjacent reliability hazard (latency spikes, connection exhaustion on serverless Neon).

More importantly for correctness: because each `resolvePrinting` call is `await`-ed sequentially inside the loop rather than batched, any transient Neon cold-start (scale-to-zero) on the first call adds full cold-start latency to *every* subsequent call. The Neon skill explicitly warns about cold-start penalties — chaining sequential queries amplifies this.

**Fix:** Collect all non-null leader/base definition IDs up front, fetch them in a single query with `inArray`, then resolve from the in-memory map:

```typescript
// After Step 2, before Step 4:
const leaderBaseIds = userDecks
  .flatMap(d => [d.leaderCardDefinitionId, d.baseCardDefinitionId])
  .filter((id): id is number => id !== null);

const leaderBasePrintings = leaderBaseIds.length > 0
  ? await db
      .select({
        cardDefinitionId: cardDefinitions.id,
        name: cardDefinitions.name,
        type: cardDefinitions.type,
        setCode: cardPrintings.setCode,
        collectorNumber: cardPrintings.collectorNumber,
        frontArtUrl: cardPrintings.frontArtUrl,
        backArtUrl: cardPrintings.backArtUrl,
      })
      .from(cardDefinitions)
      .innerJoin(cardPrintings, eq(cardDefinitions.id, cardPrintings.cardDefinitionId))
      .where(
        and(
          inArray(cardDefinitions.id, leaderBaseIds),
          eq(cardPrintings.variantType, 'Normal')
        )
      )
  : [];

const leaderBasePrintingMap = new Map(
  leaderBasePrintings.map(p => [p.cardDefinitionId, p])
);

// Step 4: replace resolvePrinting(defId) with leaderBasePrintingMap.get(defId)
const syntheticRows: typeof deckCardRows = [];
for (const deck of userDecks) {
  for (const defId of [deck.leaderCardDefinitionId, deck.baseCardDefinitionId]) {
    if (!defId) continue;
    const printing = leaderBasePrintingMap.get(defId);
    if (!printing) continue;
    syntheticRows.push({
      deckId: deck.id,
      cardDefinitionId: defId,
      quantity: 1,
      isSideboard: false,
      ...printing,
    });
  }
}
```

## Warnings

### WR-01: `updateDeck` passes raw, unvalidated `body` to DB layer when saving as draft

**File:** `src/app/api/decks/[id]/route.ts:101` (called from `src/db/queries/decks.ts:44`)

**Issue:** The PATCH handler validates the deck only when `isDraft === false`. When saving as a draft it calls `updateDeck(deckId, body)` where `body` is the raw parsed JSON from the request — including any extra or malformed fields. Inside `updateDeck`, the payload is built with explicit field guards (`if (data.name !== undefined) …`), which limits the immediate damage, but the `cards` array receives no per-element validation: `quantity` could be `0`, negative, a string, or `null`, and `isSideboard` could be any truthy value. These invalid rows are inserted directly into `deckCards`.

**Fix:** Add input validation (zod or manual guards) to the PATCH handler before forwarding `body` to `updateDeck`, regardless of draft status:

```typescript
// Validate card entries
if (Array.isArray(body.cards)) {
  for (const c of body.cards) {
    if (
      typeof c.cardDefinitionId !== 'number' ||
      typeof c.quantity !== 'number' ||
      c.quantity < 1 ||
      typeof c.isSideboard !== 'boolean'
    ) {
      return new Response('Invalid card entry', { status: 400 });
    }
  }
}
```

*(This file is not in the formal review scope but is the direct consumer of the reviewed query. Flagged because the bug manifests via `updateDeck`.)*

### WR-02: `want-list-tab.tsx` silently fails on non-200 collection fetch — collection stays empty

**File:** `src/components/decks/want-list-tab.tsx:18-29` (consumed by `deck-builder.tsx:343`)

**Issue:** The `useEffect` that fetches `/api/collection` does not check `res.ok` before calling `res.json()`. If the API returns a 500, `res.json()` will throw (the body is plain text `"Internal Server Error"`), which is caught by the `.catch` block — `collection` stays as `{}` (empty), `loading` becomes `false`, and the UI silently reports "You own all the cards for this deck." The user sees a false-positive "nothing to buy" state rather than an error message.

**Fix:**

```typescript
.then(res => {
  if (!res.ok) throw new Error(`Collection fetch failed: ${res.status}`);
  return res.json();
})
.then(data => {
  setCollection(data);
  setLoading(false);
})
.catch(err => {
  console.error('Failed to load collection:', err);
  setError(true);   // add useState<boolean> error state
  setLoading(false);
});
```

Then render an error state instead of the shortfall list when `error` is true.

### WR-03: Sideboard cards are excluded from the Want List in `deck-builder.tsx`

**File:** `src/components/decks/deck-builder.tsx:344-349`

**Issue:** The `deckCards` prop passed to `WantListTab` explicitly filters to `!c.isSideboard`:

```tsx
deckCards={[
  ...state.cards.filter(c => !c.isSideboard)   // sideboard excluded
    .map(c => ({ cardDefinitionId: c.cardDefinitionId, quantity: c.quantity })),
  ...(state.leaderCardDefinitionId ? [...] : []),
  ...(state.baseCardDefinitionId   ? [...] : []),
]}
```

Sideboard cards are legal deck components that a player may need to acquire. Excluding them produces an incomplete want list — the user could purchase all flagged cards and still be short for sideboard slots.

**Fix:** Include sideboard cards (the `WantListTab` already accepts a flat array so no interface change is needed). If you want to distinguish sideboard shortfalls visually, add a section header, but excluding them silently is wrong:

```tsx
deckCards={[
  ...state.cards.map(c => ({ cardDefinitionId: c.cardDefinitionId, quantity: c.quantity })),
  ...(state.leaderCardDefinitionId ? [{ cardDefinitionId: state.leaderCardDefinitionId, quantity: 1 }] : []),
  ...(state.baseCardDefinitionId   ? [{ cardDefinitionId: state.baseCardDefinitionId,   quantity: 1 }] : []),
]}
```

### WR-04: Duplicate `cardDefinitionId` entries possible in `WantListTab` input — shortfall undercounted

**File:** `src/components/decks/deck-builder.tsx:344-349`

**Issue:** It is possible (though currently unlikely due to the UI) for `state.cards` to contain a card that is also set as `leaderCardDefinitionId` or `baseCardDefinitionId`. The `deckCards` array is built by concatenation without deduplication. `WantListTab.shortfallCards` maps over `deckCards` with `.map(dc => …)` — it does not aggregate duplicates — so the same `cardDefinitionId` produces two separate shortfall entries displayed in the UI. Additionally, if a Leader is in `state.cards` with `isSideboard: false`, the quantity from its `deckCards` row and the synthetic `quantity: 1` are never summed.

**Fix:** Aggregate by `cardDefinitionId` before passing to `WantListTab`:

```typescript
const wantListCards = useMemo(() => {
  const totals = new Map<number, number>();
  state.cards
    .filter(c => !c.isSideboard)
    .forEach(c => totals.set(c.cardDefinitionId, (totals.get(c.cardDefinitionId) ?? 0) + c.quantity));
  if (state.leaderCardDefinitionId)
    totals.set(state.leaderCardDefinitionId, Math.max(totals.get(state.leaderCardDefinitionId) ?? 0, 1));
  if (state.baseCardDefinitionId)
    totals.set(state.baseCardDefinitionId, Math.max(totals.get(state.baseCardDefinitionId) ?? 0, 1));
  return [...totals.entries()].map(([cardDefinitionId, quantity]) => ({ cardDefinitionId, quantity }));
}, [state.cards, state.leaderCardDefinitionId, state.baseCardDefinitionId]);
```

## Info

### IN-01: `Record<string, any>` type for `updatePayload` loses type safety

**File:** `src/db/queries/decks.ts:56`

**Issue:** `updatePayload` is typed as `Record<string, any>`, defeating TypeScript's ability to catch typos in field names or wrong value types before they hit the DB. This is particularly risky because `updatePayload` is passed directly to `tx.update(decks).set(updatePayload)`.

**Fix:** Use the Drizzle inferred type for the update shape, or at minimum a more specific partial type:

```typescript
import type { InferInsertModel } from 'drizzle-orm';
type DeckUpdate = Partial<Omit<InferInsertModel<typeof decks>, 'id' | 'userId' | 'createdAt'>>;
const updatePayload: DeckUpdate = {};
```

### IN-02: `console.error` debug artifacts left in production path

**File:** `src/components/decks/deck-builder.tsx:198`

**Issue:** `console.error('Failed to save deck', err)` is present in the catch block of `handleSave`. While this is a catch block (not silent), the `err` object printed to the browser console may include stack traces or server-side error messages that could aid enumeration. In a production build, a structured error boundary or monitoring hook (e.g., Sentry) is preferable.

**Fix:** Replace with a user-visible error state (already done via `setApiErrors`) and remove or gate the `console.error` behind a dev-mode flag:

```typescript
if (process.env.NODE_ENV === 'development') {
  console.error('Failed to save deck', err);
}
```
