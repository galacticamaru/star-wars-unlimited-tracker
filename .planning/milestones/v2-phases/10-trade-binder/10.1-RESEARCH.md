## Summary

Phase 10.1 addresses a logic discrepancy between the Global Want List and the Trade Binder's "Looking For" section. While the Global Want List correctly excludes sideboard cards from its requirements, the Trade Binder's public data query in `src/db/queries/binder.ts` currently includes them. This results in inflated "Looking For" quantities when a user has cards in their sideboard that they do not own in their collection.

**Primary recommendation:** Apply a surgical patch to the `deckCards` sub-query within `getPublicBinderData` to filter for `isSideboard: false`.

### Pattern 1: Multi-Condition Filtering (Drizzle)
**What:** Combining multiple criteria in a `where` clause using `and()`.
**When to use:** When a query needs to filter by multiple columns (e.g., `deckId` AND `isSideboard`).
**Example:**
```typescript
// Source: https://orm.drizzle.team/docs/operators#and
import { and, eq, inArray } from 'drizzle-orm';

await db.select().from(table).where(
  and(
    inArray(table.id, ids),
    eq(table.status, 'active')
  )
);
```

### Anti-Patterns to Avoid
- **Client-side filtering:** Fetching sideboard cards and filtering them in JavaScript is less efficient than filtering at the database level, especially for users with many decks.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Boolean Logic | Custom string concatenation | `and()`, `eq()` | Prevents SQL injection and ensures type safety. |

## Common Pitfalls

### Pitfall 1: Missing `and()` wrapper
**What goes wrong:** Providing multiple arguments to `.where()` without an `and()` wrapper.
**Why it happens:** Some ORMs (like Sequelize) handle multiple arguments as implicit ANDs, but Drizzle requires an explicit operator.
**How to avoid:** Always wrap multiple conditions in `and()`.
