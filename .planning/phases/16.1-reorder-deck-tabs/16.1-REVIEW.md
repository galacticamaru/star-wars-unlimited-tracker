## Warnings

### WR-01: Mobile card preview bottom bar never dismisses after a tap

**File:** `src/components/decks/deck-builder.tsx:518` and `573`

**Issue:** The main deck and sideboard row elements handle `onTouchStart` to set `hoveredCard`, but there is no `onTouchEnd` or `onTouchCancel` handler. On mobile, once the user taps a card row, `hoveredCard` is set and the fixed bottom bar (lines 606–621) appears and stays visible permanently — it can only be cleared by tapping a different card row or by using keyboard focus/blur. Tapping neutral areas of the page (e.g. the header, the sideboard header, padding regions) does not dismiss the bar.

**Fix:** Add `onTouchEnd` to each interactive row to clear `hoveredCard`:

```tsx
// Main deck row (line ~519):
onTouchEnd={() => setHoveredCard(null)}

// Sideboard row (line ~574):
onTouchEnd={() => setHoveredCard(null)}
```

Add both handlers alongside the existing `onTouchStart` on each `<div ... tabIndex={0}>` row.

### WR-02: Sideboard card silently dropped from persisted state when cardMap lookup fails

**File:** `src/components/decks/deck-builder.tsx:214–229`

**Issue:** The `sideboard` memo (lines 214–229) filters out any card whose `cardDefinitionId` is not present in `cardMap`. This is the correct display behavior. However, the save payload at line 283 sends `state.cards` directly — which includes all raw card entries regardless of lookup status. The asymmetry itself is not a bug, but the `deckCounts` memo (lines 233–241) only counts non-sideboard cards and is passed to `CatalogClient`. If a card is in the sideboard and also in the main deck at the display level, the catalog badge only reflects main-deck quantity. This is consistent with the existing design (sideboard is not shown in the catalog badge), but the sideboard count display on line 555 recomputes from `sideboard.reduce(...)` rather than from `sideboardTotal` (which is already memoized at line 266–269). The inline reduce at line 555 is a redundant recomputation:

```tsx
// Line 555 — redundant; sideboardTotal is already computed at line 266
<h3>Sideboard ({sideboard.reduce((s, i) => s + i.quantity, 0)} / 10)</h3>
```

**Fix:** Use the memoized value already available:

```tsx
<h3 className="text-lg font-bold">
  Sideboard ({sideboardTotal} / 10)
</h3>
```

This eliminates the duplicate iteration and keeps display consistent with the source of truth used to enforce the 10-card cap.

### IN-01: console.error left in production error handler

**File:** `src/components/decks/deck-builder.tsx:299`

**Issue:** `console.error('Failed to save deck', err)` in the `catch` block of `handleSave` leaks internal error objects to the browser console in production. The user-visible error path already sets `apiErrors` correctly; the console call provides no additional user value in production.

**Fix:** Remove the `console.error` call, or guard it behind a development-only check:

```tsx
// Option A — remove entirely (user error is already surfaced via apiErrors):
// console.error('Failed to save deck', err);

// Option B — guard for dev only:
if (process.env.NODE_ENV === 'development') {
  console.error('Failed to save deck', err);
}
```
