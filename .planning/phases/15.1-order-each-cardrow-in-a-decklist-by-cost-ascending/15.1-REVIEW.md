## Warnings

### WR-01: `isDraft` state never synced in reducer after save — persistent false-positive dirty flag

**File:** `src/components/decks/deck-builder.tsx:270`
**Issue:** `handleSave` updates `cleanStateRef.current` to `{ ...state, isDraft }`, where `isDraft` is the boolean argument passed to the function. However, the reducer has no action that updates `state.isDraft`. This means the `isDirty` comparison (line 142) diverges whenever the `isDraft` field in `cleanStateRef` differs from `state.isDraft`.

Concrete reproduction: open a published deck (`initialDeck.isDraft = false`). Click "Save as Draft" (`handleSave(true)`). On success, `cleanStateRef.current.isDraft` becomes `true`, but `state.isDraft` remains `false`. `isDirty` immediately becomes `true` with no user-visible change. The `beforeunload` guard then fires on every subsequent page-close attempt even though nothing is actually unsaved.

**Fix:** Add a `SET_DRAFT` action to the reducer and dispatch it on successful save:

```typescript
// In DeckAction union:
| { type: 'SET_DRAFT'; payload: boolean }

// In deckReducer:
case 'SET_DRAFT':
  return { ...state, isDraft: action.payload };

// In handleSave, on success:
cleanStateRef.current = { ...state, isDraft };
dispatch({ type: 'SET_DRAFT', payload: isDraft });
```

### WR-02: Export reflects last-saved server state when deck has unsaved changes

**File:** `src/components/decks/deck-builder.tsx:333-338`
**Issue:** The Export dropdown items open `/api/decks/${state.id}/export` directly in a new tab. The server always reads from the database, so when `isDirty` is true (unsaved changes exist in memory), the export silently produces a stale file that omits all in-memory edits. There is no warning to the user.

**Fix:** Disable the export trigger or show a warning when `isDirty` is true:

```tsx
<DropdownMenuItem
  onClick={() => {
    if (isDirty && !window.confirm('You have unsaved changes. Export the last saved version?')) return;
    window.open(`/api/decks/${state.id}/export?format=melee`, '_blank');
  }}
>
  Melee (.txt)
</DropdownMenuItem>
```

### WR-03: `onTouchStart` sets `hoveredCard` with no `onTouchEnd` to clear it — mobile preview bar cannot be dismissed

**File:** `src/components/decks/deck-builder.tsx:496, 551`
**Issue:** Both the main deck rows and sideboard rows set `hoveredCard` on `onTouchStart` but have no `onTouchEnd`/`onTouchCancel` handler. On mobile, `onMouseLeave` and `onBlur` do not fire after a tap. This means the mobile fixed preview bar (lines 584-599) appears when a card is tapped and never disappears unless a different card is tapped. The bar obscures the bottom action buttons.

**Fix:** Add `onTouchEnd` and `onTouchCancel` to clear the hovered card, or use a timed auto-dismiss:

```tsx
onTouchEnd={() => setHoveredCard(null)}
onTouchCancel={() => setHoveredCard(null)}
```

### IN-02: Deck name `<input>` has no `maxLength` constraint

**File:** `src/components/decks/deck-builder.tsx:294-299`
**Issue:** The deck name input has no `maxLength` attribute. A user can type an arbitrarily long string, which will overflow the toolbar layout and may exceed any database column constraint silently (the API would return a 400 or 500 that surfaces as a generic error).

**Fix:** Add `maxLength` matching whatever the database column allows (e.g., 100 characters):

```tsx
<input
  value={state.name}
  onChange={(e) => dispatch({ type: 'SET_NAME', payload: e.target.value })}
  maxLength={100}
  className="..."
  placeholder="Deck Name"
/>
```
