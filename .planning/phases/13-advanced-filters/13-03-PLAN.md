<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL → CatalogClient | `?owned=true` URL parameter is parsed by nuqs `parseAsBoolean`; only true/false accepted |
| CatalogClient → filterCards | ownedOnly boolean and collection map; both are client-side in-memory values |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-03-01 | Spoofing | URL ?owned param | accept | nuqs `parseAsBoolean` rejects non-boolean URL values — invalid input falls back to the default (false). The filter is display-only; server-side data access is unchanged and protected by existing auth middleware. |
| T-13-03-02 | Information Disclosure | Client-side collection data | accept | Collection data is already fetched from `/api/collection` which enforces session auth. Owned-only filtering reveals no new data — it hides cards, it does not expose them. |
| T-13-03-03 | Denial of Service | URL param manipulation | accept | No server load change — filtering is entirely client-side. Malicious URL params are silently ignored by nuqs. |

## ASVS Categories Verified N/A

| ASVS Category | Rationale |
|---------------|-----------|
| V2 Authentication | No new auth surfaces; `isAuthenticated` is read from existing session, not computed here |
| V3 Session Management | No session changes |
| V4 Access Control | The toggle is display-only. All existing API auth guards remain unchanged. A logged-out user who manually sets ?owned=true in the URL sees 0 results (empty collection) — they see no data they shouldn't |
| V5 Input Validation | nuqs parseAsBoolean validates the URL param; non-boolean values default to false |
| V6 Cryptography | Not applicable |
</threat_model>
