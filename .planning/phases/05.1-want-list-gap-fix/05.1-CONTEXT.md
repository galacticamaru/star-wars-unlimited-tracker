# Phase 5.1: Want List Gap Fix - Context

**Gathered:** 2026-05-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 5.1 is a surgical bug fix. It extends the want list computation to include the Leader and Base cards that are stored as FK columns on the `decks` table (`leaderCardDefinitionId`, `baseCardDefinitionId`) but are currently invisible to both want list views.

**Scope:**
- Fix per-deck want list (`WantListTab`) to include leader/base shortfalls
- Fix combined want list (`getDeckCardsForUser()` + `/api/want-list`) to include leader/base shortfalls
- No new UI, no new routes, no schema changes

**Out of scope:**
- Any UI improvements beyond what's needed to surface the missing cards
- Sideboard or variant handling
- Export or sharing features (v2)

</domain>

<decisions>
## Implementation Decisions

### Incomplete Deck Edge Cases
- **D-01:** When `leaderCardDefinitionId` is null (deck has no leader assigned), do NOT show a shortfall for that slot. Skip it silently â€” deck is a work-in-progress and "missing card" is confusing before a choice is made.
- **D-02:** Same rule for base: null `baseCardDefinitionId` â†’ skip, no shortfall emitted.
- **D-03:** This applies to both the per-deck want list (WantListTab) and the combined want list (/api/want-list). Consistent throughout.

### WantListTab Prop Interface
- **D-04:** Extend the existing `deckCards` array at the call site in `deck-builder.tsx` â€” inject leader and base as `{cardDefinitionId, quantity: 1}` entries before passing the prop. No change to WantListTab's interface.
  ```tsx
  deckCards={[
    ...state.cards.filter(c => !c.isSideboard)
      .map(c => ({ cardDefinitionId: c.cardDefinitionId, quantity: c.quantity })),
    ...(state.leaderCardDefinitionId ? [{ cardDefinitionId: state.leaderCardDefinitionId, quantity: 1 }] : []),
    ...(state.baseCardDefinitionId   ? [{ cardDefinitionId: state.baseCardDefinitionId,   quantity: 1 }] : []),
  ]}
  ```
- **D-05:** The existing empty-state check (`if (deckCards.length === 0)`) is correct as-is â€” it fires only when deck has no leader, no base, and no main deck cards. No change needed.

### getDeckCardsForUser() â€” Combined Want List
- **D-06:** Merge leader/base into the same result set returned by `getDeckCardsForUser()`. Emit synthetic rows with `quantity: 1, isSideboard: false` for each non-null leader/base on every deck. Uniform row shape â€” the route and aggregation logic in `/api/want-list` need no changes.
- **D-07:** Join `cardPrintings` for leader/base rows using the same `variantType = 'Normal'` filter already applied to `deck_cards` rows. Consistent image URL resolution throughout.
- **D-08:** Decks with null leader or base contribute no synthetic row for that slot (consistent with D-01/D-02).

</decisions>
